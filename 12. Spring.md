- Service discovery and load balancing
    Service discovery helps microservices find each other dynamically without hardcoded URLs. Services register themselves in a registry like Eureka.

    In client-side load balancing, the client service fetches all available instances from Eureka and chooses one instance using Spring Cloud LoadBalancer. The load balancing logic runs inside the client.

    In server-side load balancing, the client sends requests to a dedicated load balancer such as Nginx, AWS ELB, or Kubernetes Service. The load balancer selects the target instance and forwards the request.

    Spring Boot microservices commonly use Eureka for service discovery and Spring Cloud LoadBalancer for client-side load balancing

- ISTIO MESH

    In Istio, application services do not perform load balancing themselves. Each pod has an Envoy sidecar proxy. Service discovery information is distributed to Envoy by Istio's control plane, and Envoy performs load balancing between healthy service instances. This is effectively proxy-based/server-side load balancing and enables advanced traffic management features such as retries, circuit breaking, canary releases, and traffic splitting without changing application code
    In a Kubernetes + Istio environment, Eureka is usually not required. Kubernetes provides service discovery, while Istio's Envoy sidecars provide load balancing, retries, circuit breaking, and traffic management. So Eureka is typically replaced by Kubernetes service discovery plus Istio's control plane, not by Envoy alone. Envoy consumes service information; it does not act as a service registry by itself
