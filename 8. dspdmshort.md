# DSPDM Services — Master Summary

## 🏗️ What is DSPDM?

**DSPDM** (Dynamic Sub-Surface Data Management) is a **metadata-driven, multi-tenant data management platform** for the **oil & gas industry**.

- Manages: Wells, Wellbores, Fields, Areas, Platforms, Pipelines, Equipment, Production Volumes, and 100+ more entities
- **Core Idea**: Zero static entity classes — the data model lives in the database itself, driven by metadata tables
- **Stack**: Java 17, JAX-RS (Jersey), Grizzly HTTP, PostgreSQL, Redis, Teiid VDB, Bitronix JTA, Apache POI

---

## 🎯 The Single Most Important Design Decision

> **No JPA. No Hibernate. No `@Entity` classes. Ever.**

```
Traditional:  WellEntity.java → hardcoded → redeploy to add field
DSPDM:        BUSINESS_OBJECT_ATTR row → metadata → zero code change
```

`DynamicDTO` extends `TreeMap<String, Object>` with `CASE_INSENSITIVE_ORDER`.  
Every row from any table — WELL, R_WELL_TYPE, metadata — is a `DynamicDTO`.

## 🔄 Complete Read Flow (POST /common)

```
1. POST /common  {"boName":"WELL", "pagination":{"pages":[1],"recordsPerPage":50}}
2. AuthenticationFilter → JWT RS256 verify via Keycloak JWKS (cached by kid)
3. MainserviceImpl.getCustomResponsePost()
4. BusinessObjectBuilder.build() → BOQuery JSON → BusinessObjectInfo
   ├─ WAF bypass decoding ($having$ → having)
   ├─ Pagination, filters, orderBy, joins
5. DTOHelper.convertDataTypeForFilters() → cast filter values
6. dynamicReadService.readWithDetails()
7. AbstractDynamicDAOImplForRead.buildSelectSQL()
   → SELECT well_id, well_name FROM well WHERE tenant_id=? LIMIT 50 OFFSET 0
8. PreparedStatement.executeQuery() → ResultSet → List<DynamicDTO>
9. UnitHelper.unitConvertAfterRead() → feet→meters per user template
10. DSPDMResponse → JSON
```

## 🔄 Complete Write Flow (POST /save)

```
1. POST /save  {"WELL": {"data": [{"WELL_NAME":"ABC-01"}]}}
2. JWT verify → ExecutionContext(tenantId, userName, permissions)
3. DTOHelper.recursiveDateUnitConverion() → convert values BEFORE save
4. DTOHelper.buildBONameAndBOListMapFromRequestJSON()
   ├─ Read metadata from DB (column types, lengths, FK refs)
   ├─ ConcurrentUtils.concurrentByGroup(10) → 10 DTOs built in parallel
   ├─ validateAndConvertDataType() → String → correct Java type
   ├─ validateReferenceAttrValue() → live DB count() for FK check
   └─ JsonbValidationUtils.validateJsonbData() → JSONB mandatory fields
5. dynamicWriteService.saveOrUpdate()
   ├─ beginTransaction() (Bitronix JTA / XA)
   ├─ getNextFromSequence() → PK
   ├─ INSERT INTO well (...) VALUES (?,?) via executeBatch()
   └─ commitTransaction()
6. SearchHelper.createSearchIndexForSavedOrUpdatedRecord()
   ├─ DELETE old BO_SEARCH entries
   ├─ Re-read saved record
   └─ INSERT new BO_SEARCH (JSONB)
7. DSPDMResponse → JSON
```

## 💾 Three-Layer Caching

```
Layer 1: In-Memory Metadata (DynamicDAOFactory)
  → ALL BO schemas loaded at startup → zero DB hits for column info
  → Refresh: GET /refreshMetadata

Layer 2: Redis Distributed Cache (DSPDMRedisCacheUtils)
  → Reference data: R_WELL_TYPE, R_STATUS, etc.
  → User permissions (Keycloak roles → BO permissions)
  → Key pattern: <tenantId>:<boName>
  → Cache-aside: check Redis → miss → DB → populate Redis

Layer 3: Unit Conversion In-Memory (MSMHelper)
  → Entire unit catalog in HashMaps at startup
  → Refresh: GET /refreshUnitConversionCache
```

## 🔍 Search Architecture (BO_SEARCH)

### Why Not ILIKE on Columns?

```sql
-- BAD (slow, 10M+ records): multiple joins + ILIKE = no index
SELECT * FROM well w JOIN field f ON ...
WHERE w.well_name ILIKE '%ABC%' OR f.field_name ILIKE '%ABC%'

-- GOOD (DSPDM): single table GIN index
SELECT bo_name, pk_bo_attr_values FROM bo_search
WHERE search_jsonb::text SIMILAR TO '%ABC%'
AND bo_name='WELL' AND tenant_id=?
```

### BO_SEARCH Table
```sql
CREATE TABLE bo_search (
    bo_search_id     SERIAL PRIMARY KEY,
    bo_name          VARCHAR,
    pk_bo_attr_values VARCHAR,
    search_jsonb     JSONB,         -- searchable fields (GIN indexed)
    object_jsonb     JSONB,         -- complete record
    tenant_id        VARCHAR
);
CREATE INDEX ON bo_search USING GIN(search_jsonb);
```

### Two-Phase Search
```
Phase 1: GIN search on bo_search → returns PKs (fast, single table)
Phase 2: SELECT by PK from main table → returns full records
```

### Search Index Update on Save
```
Every save → SearchHelper:
  DELETE old BO_SEARCH by PK
  Re-read fresh record from DB
  INSERT new BO_SEARCH with regenerated JSONB
```

## 🔐 Security System

```
1. AuthenticationFilter (@Priority AUTHENTICATION)
   → Extracts Bearer JWT
   → KeycloakJWTVerifier.verifyToken()
     → Fetches RSA key from Keycloak JWKS (cached by kid in ConcurrentHashMap)
     → RS256 signature verification
   → Extracts tenantId, userName, roles → ExecutionContext

2. Authorization (programmatic, no @RolesAllowed)
   → ApiPermission enum: OPEN, VIEW, EDIT, DELETE, SYSTEM_ADMIN
   → Checked per endpoint in getCurrentUserExecutionContext()

3. Multi-tenancy
   → TenantContext in ExecutionContext
   → Every query routed to tenant schema
   → Redis keys: <tenantId>:<boName>
   → Row Level Security on PostgreSQL connection

4. Input Sanitization
   → WAF bypass: $having$ → having (Azure WAF compatibility)
   → All values via PreparedStatement (no concatenation)
   → JSONB injection: blocked chars ' " - _ \ ( ) [ ] ? + *
```

## 📊 PVS (Production Volume Summary)

```
Problem:  GROUP BY on 50M+ daily production records = seconds
Solution: Pre-aggregated PRODUCT_VOLUME_SUMMARY table

Special technique — AMENDMENT_SEQ_NO:
  Same data can have multiple versions (amendments 1, 2, 3...)

  DynamicJoin (subquery):
    SELECT ..., MAX(AMENDMENT_SEQ_NO) as max_amendment_seq_no
    FROM PRODUCT_VOLUME_SUMMARY
    GROUP BY START_DATE, REPORTING_ENTITY_ID, ...

  Joined back to main table:
    pvs1.AMENDMENT_SEQ_NO = pvs2.max_amendment_seq_no

  Result: ONLY latest amendment per date+product+flow combination

Filter routing:
  ASSOCIATED_OBJECT_ID filter → goes to REPORTING_ENTITY join (outer)
  All other filters → go INSIDE subquery (MAX calculated on filtered scope)

Table hierarchy:
  WELL_VOL_DAILY      → 50M+ records, real-time, single well queries
  WELL_VOL_MONTHLY    → 2M+ records, multi-well monthly reports
  WELL_VOL_YEARLY     → 200K+ records, historical trends
  MV_PROD_ENTITY_MONTH → 500K+ pre-calculated, complex hierarchy queries

Auto-routing:
  Single well + ≤31 days   → WELL_VOL_DAILY     (10-50ms)
  Multi-well + ≤365 days   → WELL_VOL_MONTHLY   (100-500ms)
  Country + multi-year     → Materialized View  (500-2000ms)
```

## ❗ Design Trade-offs

| Decision | Benefit | Trade-off |
|----------|---------|-----------|
| Dynamic DTO (no entities) | Runtime schema changes, zero redeployment | No compile-time type safety |
| Raw JDBC (no ORM) | Full SQL control, dynamic generation | Complex builder code ~6000 lines |
| Explicit ExecutionContext | Testable, no hidden state | Verbose — every method has `ExecutionContext` param |
| Deep DAO inheritance (10 levels) | Clean capability separation | Hard to navigate, changes ripple upward |
| Single shared DB + RLS | Cost-efficient, one schema | Cross-tenant leak risk if RLS misconfigured |
| Metadata-driven validation | Self-describing, no code changes | Runtime errors instead of compile-time |

## 🔑 Key Patterns Summary

```
1. Metadata-Driven:     Schema in DB → DAO built at runtime → SQL generated dynamically
2. Dynamic DTO:         TreeMap<String,Object> with CASE_INSENSITIVE_ORDER
3. Explicit Context:    ExecutionContext passed to EVERY method (not ThreadLocal)
4. Three-Class Delegate: Interface → Outer (tx/exception) → Inner (pure logic)
5. Cache-Aside:         Check Redis → miss → DB → populate Redis
6. Two-Phase Search:    GIN on JSONB → PKs → fetch full records
7. DELETE+INSERT Index: BO_SEARCH always fully regenerated on save (never partial update)
8. Parallel DTO Build:  10 records processed concurrently per request
9. Row-Level Tenant:    tenantId on every connection + Redis key prefix
10. WAF Bypass Tokens:  $having$ → having, $roundbracketopen$ → (
```

## 🎯 One-Liner Explanations for Common Questions

**"How do you add a new table/entity?"**
> Insert row in `BUSINESS_OBJECT` + rows in `BUSINESS_OBJECT_ATTR` → call `GET /refreshMetadata` → immediately available via REST API. Zero code change.

**"How does pagination work?"**
> `BusinessObjectInfo.pagination` → `OFFSET ? ROWS FETCH NEXT ? ROWS ONLY` in generated SQL. Client specifies `pages[]` + `recordsPerPage`.

**"How is SQL injection prevented?"**
> All user values bound via `PreparedStatement.setObject()`. Zero string concatenation of user input. Column/table names validated against metadata registry.

**"How do concurrent requests not interfere?"**
> DAOs are stateless (all state in method params). `ExecutionContext` is per-request. Metadata shared via `ConcurrentHashMap` (reads only after startup).

**"What happens on cache miss?"**
> Redis miss → DB query → populate Redis → return. Metadata miss → `DynamicDAOFactory.refreshMetadata()` → rebuild DAO registry.

**"How is multi-tenancy enforced?"**
> JWT → tenantId → `TenantContext`. Every DB query routes to tenant's schema. Every Redis key prefixed with tenantId. PostgreSQL RLS as second layer.

**"How does equipment data work differently?"**
> Equipment uses EAV (Entity-Attribute-Value) pattern. `EQUIPMENT_SPEC` stores rows like `(equipment_id, spec_name, value)`. API converts to column-oriented `DynamicDTO`.

**"Why not Spring Boot?"**
> JAX-RS (Jersey) + Grizzly HTTP server. Service locator pattern (`.getInstance()`) instead of CDI injection. Bitronix JTA for XA transactions across multiple data sources.

**"How are transactions managed?"**
> JTA (Java Transaction API) via Bitronix. `BaseTransactionalDelegate.beginTransaction()` → business logic → `commitTransaction()` or `rollbackTransaction()`. Supports XA (two-phase commit).

**"How does PVS return only the latest amendment?"**
> Self-join with subquery: `SELECT MAX(AMENDMENT_SEQ_NO) GROUP BY START_DATE, REPORTING_ENTITY_ID, ...` → join back on `AMENDMENT_SEQ_NO = max_amendment_seq_no`. Only latest version per date+product+flow returned.
