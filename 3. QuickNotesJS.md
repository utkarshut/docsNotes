- var vs let vs const
  var - Accessible anywhere in the function, even outside blocks like {} in if or for
  Hoisted to the top of the function and initialized as undefined.
  So you can access it before the line where you declare it, but it will be undefined

  let - block-scoped,  where it’s declared (e.g., inside if, for, {}).
  Hoisted but not initialized (temporal dead zone).
  Accessing it before declaration throws a ReferenceError.

  const - block-scoped (like let), For objects/arrays, the contents can change, but you can’t point the variable to a new value 
  Hoisting: Like let, in the temporal dead zone until declaration

- Closure
  A closure is the combination of a function bundled together with references to its surrounding state (the lexical environment). In other words, a closure gives you access to an outer function’s scope from an inner function

    const closureExample = outer(); // outer finishes, but message is still kept
    closureExample(); // prints: "Hello"

    inner is a closure: it “remembers” message from outer even after outer has returned.

- Event Loop
  JavaScript is single-threaded, but it can handle asynchronous operations (like setTimeout, fetch, DOM events) without blocking. The event loop is what makes this possible.

  The event loop continuously checks:
        Is the call stack empty?
        Are there tasks in the queues (microtask queue, callback/task queue)?
        If the call stack is empty, it moves tasks from the queues to the call stack for execution.

- Call Stack 
    The call stack is a LIFO data structure that JavaScript uses to track function execution. It keeps track of which function is currently running and what functions called it

- Promise
    Promise in JavaScript — clear, interview-ready explanation
    A Promise is an object that represents the eventual completion (success) or failure (error) of an asynchronous operation and its resulting value.

    It solves callback hell and makes async code cleaner and more composable

    A Promise has three states
    Pending	Initial state; neither fulfilled nor rejected
    Fulfilled	Async operation completed successfully
    Rejected	Async operation failed

- Microtask/ Macrotask
    The main difference between microtasks and macrotasks is priority: the event loop always processes all microtasks first, then only one macrotask, and repeats
    Microtasks are for urgent work ---> Promise callbacks: .then() .catch() .finally(),,queueMicrotask(callback), MutationObserver,
                                        process.nextTick() in Node (very high priority microtask-like)
    Macrotasks---> setTimeout, setInterval, setImmediate, User input events (clicks, keypresses), UI rendering events

- this 
  The this keyword is a reference to an object that depends on how a function is called, not where it’s defined. Its value is determined by the call site

   Rules for this in different contexts
   a) Global context
      console.log(this); // window (in browser)
   b) Inside a regular function (non-strict mode)
       function foo() {
        console.log(this);
        }
        foo(); // window (in browser)
    **In strict mode ('use strict'), this is undefined.**
        'use strict';
         function bar() {
           console.log(this);
         }
         bar(); // undefined
    c) Inside an object method
        const person = {
            name: "Alice",
            greet() {
                console.log("Hello, I'm", this.name);
            }
        };

        person.greet(); // "Hello, I'm Alice"
        // this.name resolves to person.name
       In a method, this is the object before the dot (the caller).  **IMPORTANT**
    e) Inside event handlers
       button.addEventListener('click', function () {
        console.log(this); // the button element
        });
    f) Arrow functions — they don’t have their own this
       const person = {
        name: "Alice",
        greet() {
            function regular() {
            console.log("Hello, I'm", this.name);
            }
            regular(); // this is undefined (strict) or window (non-strict)
        }
        };

- map vs object
  Objects are ideal for representing structured JSON-like data with known properties. Maps are better for dynamic key-value 
  collections because they support any key type, provide efficient insertion and lookup operations, preserve insertion order, 
  expose a size property, and avoid prototype-related issues. In large-scale applications such as hierarchy caching or lookup tables, I generally prefer Map over Object
  Key Types -> Because object keys become strings.
  Object Keys -> Map can use objects as keys.
  Size -> Object.keys(obj).length for object , map.size direct
  Iteration -> const [key,value] of map, Object.entries(obj)
  Performance -> Insert Delete Lookup -> Map better
  Ordering -> Map preserves insertion order.

- interface and type
Use interface mainly for object shapes / class contracts,
Use type for unions, intersections, and more advanced type tricks

interface User {
  name: string;
  age: number;
}

type UserType = {
  name: string;
  age: number;
};
**You can also extend both:**
interface Admin extends User {
  role: "admin";
}

type AdminType = UserType & {
  role: "admin";
};

1. a) Interfaces are open; types are closed
 
    Interface can be merged (declaration merging).

    Type cannot be re-declared with the same name.
    interface Animal {
        name: string;
    }

    interface Animal {
        age: number;
    }

    const dog: Animal = { name: "Rex", age: 5 }; // OK  **IMPRORTANT As BOTH ANIMAL MERGED**

    type Vehicle = {
        brand: string;
    };

    type Vehicle = {
        model: string; // ❌ Error: Duplicate identifier 'Vehicle'   **IMPRORTANT As BOTH VEHICLE CANT MERGED**
    };

    type is more general – it can represent:

    Unions, Intersections, Primitives, Tuples, Function types, which interfaces cannot directly.
    Examples:

    type Status = "success" | "error" | "loading";  // union

    type Point = { x: number } & { y: number };     // intersection

    type ID = string | number;                      // primitive union

    type Tuple = [number, string];                  // tuple   **IMPRORTANT**

    type Fn = (a: number, b: number) => number;     // function type

  2. c) Classes can implement both  **IMPRORTANT**
