-  Why can't static methods be overridden?
    Method overriding requires runtime polymorphism, but static methods are resolved at compile time because they belong to the class rather than the object and Because a static method does not depend on object data.. Therefore, static methods cannot be overridden; they are hidden instead.
    class Animal {
        static void sound() {
            System.out.println("Animal");
        }
    }

    class Dog extends Animal {
        static void sound() {
            System.out.println("Dog");
        }
    }

    Now,

    Animal a = new Dog();
    a.sound();

    What will it print?

    Output:

    Animal

    Compare with Instance Methods
    class Animal {
        void sound() {
            System.out.println("Animal");
        }
    }

    class Dog extends Animal {
        @Override
        void sound() {
            System.out.println("Dog");
        }
    }

    Now

    Animal a = new Dog();
    a.sound();

    Output

    Dog

    Why?

    Because instance methods are resolved at runtime.

-  CASCADE 
    In JPA and Hibernate, cascading is a feature that allows state changes to automatically propagate from a parent entity to its associated child entitie
    PERSIST: When you save (EntityManager.persist()) a parent entity, all associated new child entities are automatically saved to the database.REMOVE: When you delete a parent entity, all associated child entities are automatically deleted from the database.

    Candidates often confuse CascadeType.REMOVE with orphanRemoval=true. If you use CascadeType.REMOVE and delete a Parent, the Children are deleted. However, if you simply pull a Child out of the Parent's list (parent.getChildren().remove(child)), CascadeType.REMOVE does nothing. The child becomes an orphan database row. To automatically delete that disconnected child from the database, you must explicitly set orphanRemoval=true

        @OneToMany(mappedBy = "parent", cascade = CascadeType.ALL, orphanRemoval = true)
        private List<Child> children = new ArrayList<>();

-  Memory Leak Happens in Java
   Java has automatic garbage collection (GC), but memory leaks still happen when:

   Object created → Reference exists → GC keeps it alive
   Object no longer needed → Reference still exists → GC CANNOT remove it → Memory leak

   Common Causes of Memory Leaks in Java
   1. Static Collections That Never Clear - Static collections live for the entire application lifecycle. 
      Objects added are never garbage collected even if unused
   2. Resources like file handles, database connections, and streams stay open, consuming memory and 
      system resource
   3. Improper equals() and hashCode() - Without proper equals() and hashCode(), HashSet/HashMap can keep 
      adding duplicates, growing indefinitely
      class User {
            String name;
            
            // ❌ Missing equals() and hashCode()
        }

        Set<User> users = new HashSet<>();
        User u1 = new User("John");
        User u2 = new User("John");
        users.add(u1);
        users.add(u2);
    4. ThreadLocal Variables Not Removed
       private static ThreadLocal<User> userHolder = new ThreadLocal<>();

        public void setUser(User user) {
            userHolder.set(user);  // Never removed
        }
    5. Growing Caches Without Limits

- Circuit breaker
    Circuit Breaker is a resilience pattern used to prevent cascading failures in distributed systems. When a downstream service starts failing or responding slowly, the circuit breaker temporarily stops forwarding requests and fails fast instead of consuming application resources. In Java applications, we commonly implement it using Resilience4j with Spring Boot. We configure failure thresholds, open-state duration, and fallback methods to provide graceful degradation while protecting thread pools and maintaining overall system stabilit

    Timeout = Don't wait forever
    Retry = Try again
    Circuit Breaker = Stop calling unhealthy service
    Bulkhead = Don't let one service consume all resources
